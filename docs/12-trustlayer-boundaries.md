# TrustLayer 第一阶段边界说明

更新时间：2026-04-20

## 一句话定义

`TrustLayer` 当前这一阶段不是“全面解决 Prompt Injection 的最终产品”，而是一个 **面向高确定性风险的第一阶段控制面**。

它的目标很明确：

- 先把最容易出事故的入口收住
- 先把最容易造成真实损失的外发行为卡住
- 先把关键证据链记录下来

它不承诺解决所有语义操纵、所有认知漂移、所有本地执行风险。

---

## 1. TrustLayer 第一阶段明确承诺什么

当前这套 `TrustLayer` 第一阶段明确承诺四类能力。

### 1.1 输入侧基础治理

它会优先处理高确定性的输入风险：

- 隐藏 HTML / 注释 / 隐藏 footer
- 超长输入裁剪
- 第三方工具返回值默认按不可信处理

它能证明：

- 某些恶意内容在进入 Agent 前就已经被剥离或打标
- 一部分风险不会再进入后续动作链

### 1.2 外发侧高确定性控制

它会优先拦最容易造成真实事故的外发：

- secret 外发
- 明确 PII 外发
- 新域名外发
- 超大 payload 外发

它能证明：

- 有些请求会直接 `block`
- 有些请求会被拉进 `review_required`
- 有些正常请求仍然可以 `allow`

### 1.3 独立审计与回放

它会记录：

- 输入进入
- 策略命中
- 外发尝试
- 审批摘要
- 最终决策

它能证明：

- 事故链路不是黑盒
- 多轮流程和最终危险动作可以回放

### 1.4 审批可见性增强

它不会只返回一串标签，而会生成最小审批摘要，并把高风险请求排进审批队列。

它能证明：

- 风险请求可以被更清楚地展示给人
- `block` 和 `review_required` 可以被优先排序
- “看起来合理”的请求不一定还能继续伪装成低风险

---

## 2. TrustLayer 第一阶段明确不承诺什么

这一页很重要，因为它决定了产品不会被误卖成“什么都能防”。

当前 `TrustLayer` 第一阶段明确不承诺下面这些能力：

### 2.1 不承诺完整认知防御

它不承诺模型一定不会被带偏。

它能做的是：

- 缩小暴露面
- 给输入打标
- 在最终高风险外发处收口

它不能做的是：

- 真正理解所有语义级操纵
- 真正判断模型是不是已经在内部推理里偏了

### 2.2 不承诺本地执行控制

它不覆盖：

- 本地文件系统最小权限
- 命令执行隔离
- 运行时沙箱
- 插件本地安装与升级治理

这些能力属于后续 `SDK / Sidecar` 或 `Managed Runtime` 范围。

### 2.3 不承诺完整审批工作流

它现在有：

- 审批摘要
- 审批队列

但它没有：

- 真正的批准 / 拒绝动作流
- 审批状态机
- 审批人行为建模
- 审批疲劳的人因量化研究

### 2.4 不承诺覆盖低信号语义外泄

这是最关键的一条非承诺。

如果一个请求：

- 不直接出现邮箱、手机号、secret 这类高确定性信号
- 不明显命中新域名 / 大 payload / 明确敏感字段
- 看起来像普通业务摘要或普通工作流延续

那么当前 `TrustLayer` 第一阶段可能不会拦住。

## 3. 当前已验证能力边界

当前通过场景化对照、基础测试和评估样本，已经比较清楚地验证了下面这些边界。

### 3.1 当前明确能打的风险

- 字面 secret
- 明确邮箱 / 手机号类 PII
- 新域名外发
- 超大 payload 外发
- 隐藏 HTML / 注释 / hidden footer
- 不可信工具返回值
- 一部分多轮慢漂移后的最终危险外发
- 一部分审批社工包装请求

### 3.2 当前明确打不到或打不稳的风险

已知 blind spots 包括：

- 只描述“12 个客户联系人”，但不出现字面 PII
- `alice at example dot com` 这类弱混淆联系方式
- 看起来像普通业务摘要的低信号外发
- 更深层的语义操纵和目标漂移

这些不是“未来也许会有”的问题，而是当前样本已经证明存在的边界。

---

## 4. 为什么这些边界是合理的

这不是偷懒，而是产品分期。

如果第一阶段什么都承诺，最后通常会变成两种坏结果：

1. 演示很好看，但上线后发现误报和漏报都没法解释
2. 所有人都以为系统能防“所有 Prompt Injection”，结果在它本来就不擅长的地方被打穿

所以当前更合理的产品表达应该是：

**TrustLayer 第一阶段专注高确定性风险，不假装自己已经具备完整语义理解能力。**

这反而会让后续路线图更清晰：

- `TrustLayer 第一阶段`：高确定性风险治理 + 外发收口 + 审计可见性
- `Next`：更细的策略、更多规则、更多样本、更强审批面
- `Later`：真正更深的 runtime 控制和更强的语义检测

---

## 5. 什么时候应该升级到下一阶段

如果客户遇到下面这些需求，就不该继续只停在 `TrustLayer` 的第一阶段：

### 升级到更强规则 / 更强检测

适用场景：

- 已知盲点开始影响评估结果
- 误报 / 漏报需要更细调
- 客户开始要求更难的 PII / secret 识别

### 升级到 `SDK / Sidecar`

适用场景：

- 需要真正控制上下文构建
- 需要“先提议再执行”
- 需要更细的工具调用治理

### 升级到 `Managed Runtime`

适用场景：

- 需要本地执行控制
- 需要文件系统和命令沙箱
- 需要真正把宿主机能力边界做硬

---

## 6. 最适合对外讲的话

如果要把这页压成几句话，我建议就这么讲：

**TrustLayer 第一阶段不是一个“什么都能防”的万能层。**

**它优先解决的是最容易造成真实事故的高确定性风险：隐藏输入、不可信工具输出、明显敏感外发、新域名外发和可回放的审批收口。**

**对于低信号、弱混淆、深语义操纵这类问题，TrustLayer 第一阶段会诚实地把它们留在边界之外，而不是假装已经解决。**
